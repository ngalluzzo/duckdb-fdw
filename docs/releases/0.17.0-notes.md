# duckdb-fdw 0.17.0 release notes

Version 0.17.0 adds bounded reactive rate-limit handling, accepted by
[RFC 0025](../rfcs/0025-enable-bounded-reactive-rate-limit-handling.md). A
connector author can declare how a proved read-only REST or GraphQL operation
reacts to selected complete remote quota responses by publishing a
package-major `duckdb_api/v3` generation. Existing v1 and v2 packages preserve
their prior behavior.

## Declare a v3 policy

An eligible operation may declare statuses, `fail`, `wait`, or
`wait_if_deadline_allows`, a stable operation family, an explicit principal
scope, closed guidance field formats, optional remaining and remote-bucket
roles, and bounded attempt/delay/wait maxima. See the complete package under
[`examples/rate-limit-v3`](../../examples/rate-limit-v3).

Every field name is a canonical lower-case HTTP token. Waiting uses only
`retry_after`, non-negative delta seconds, or Unix seconds; there is no unit
inference or general response-header map. Authorization statuses cannot be
reclassified as quota responses. Adding, removing, or changing the policy
requires a package-major version.

## Safety boundary

Runtime reacts only after a fully received response with a declared status.
The existing replay-safe retry loop remains the sole attempt loop, and the
per-step authority is the maximum—not the sum—of ordinary retry and rate-limit
attempts. Partial responses, exposed rows, malformed guidance, excessive
delay, budget or deadline exhaustion, repeated immediate guidance, remote
bucket drift, queue saturation, and shutdown are terminal.

One executor-local bounded FIFO service coordinates equal exact destination,
package-major, operation-family, principal, and remote-bucket identities with
one in-flight permit per key. Opaque credential authority is used directly;
secret names, values, and revision hashes are not key surrogates. Independent
keys and DuckDB database instances do not block one another. Cancellation and
close wake queued work without allowing overlapping same-key transports.

Diagnostics separately report retry delay, rate-limit waiting, remote
transport time, quota event/wait counts, current wait state, and a closed
redacted reason. Response-field values, remote bucket text, opaque identities,
URLs, and timestamps are never rendered.

## Compatibility and migration

- V1 remains one-attempt; v2 retains its existing bounded retry behavior.
- Moving any package from v1 or v2 to v3 requires a package-major version,
  even when no operation declares `rate_limit`.
- A v3 package without a rate-limit block behaves like its v2 source.
- Remaining-quota values never proactively pace successful requests.
- Distributed coordination, caching, circuit breaking, parallel pages, and
  author-defined parsers remain out of scope.

## Evidence

The deterministic matrix covers schema and compilation rejection, exact
plan/explanation propagation, retry/rate-limit budget algebra, all guidance
formats, absolute-time conversion, malformed and conflicting fields,
same-key FIFO serialization, independent principals and keys, remote-bucket
stability, cancellation, queue bounds, executor close, REST and GraphQL page
atomicity, and DuckDB relational equivalence after successful recovery.
