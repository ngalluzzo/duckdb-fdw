# 0.1.0 release evidence runbook

This runbook transfers the accepted first-query build, compatibility, and
release-evidence practice to Query Experience. RFC 0001 and the propagated
architecture, connector, and runtime contracts define behavior; this document
defines how to reproduce its evidence.

For the current product capability, limitations, and ordinary `make demo`
workflow, read the [0.1.0 release notes](0.1.0-notes.md). The reusable
developer cell is intentionally separate from the fresh and tagged evidence
procedures below.

## Evidence cells

The supported product cell is DuckDB 1.5.4 at commit `08e34c447b`, platform
`osx_arm64`, on macOS 26.5.1 Apple Silicon arm64 with Apple clang 17 in C++11
mode, CMake 4.1.2, and Ninja 1.13.0. The artifact is source-built and loaded
directly with unsigned extension loading enabled.

`linux_amd64-sanitized` is a safety-evidence cell, not a supported product
platform. It runs the same production sources under AddressSanitizer and
UndefinedBehaviorSanitizer on a native Linux x86_64 executor inside the exact
Ubuntu image digest recorded in `release/0.1.0/pins.json`.

## Pre-tag product evidence

Use a path that does not exist. The runner refuses reused build state and
bootstraps checksummed CMake and Ninja binaries:

```sh
scripts/run-native-product-tests.sh /absolute/new/build-root release
```

The runner verifies source identities, fetches the exact DuckDB template and
submodule commits and trees, compiles the loadable and static test targets,
runs the private contract host and SQLLogicTests, rejects leaked test controls,
and executes both the public artifact oracle and the product-facing first-query
demo contract in the pristine pinned Python host from an isolated directory.
`make verify PROFILE=release` is a convenience wrapper around this runner that
allocates a new build root automatically.

## Authoritative product gate

After the reviewed release commit is tagged exactly `v0.1.0`, run:

```sh
scripts/run-0.1-release-gate.sh \
  /absolute/new/build-root \
  /absolute/new/evidence-root
```

The gate requires a clean worktree and `v0.1.0^{commit} == HEAD`, materializes
that commit into a detached source snapshot, and builds only tracked bytes from
the snapshot. It checks every version, content, dependency, and tool identity,
then copies the candidate artifact into the evidence root before either pristine
host loads it. DuckDB 1.5.4 must prove the accepted behavior and DuckDB 1.5.3
must reject those exact selected bytes before registration. The gate then writes
and re-verifies the artifact/manifest/checksum set and observes the dirty,
ignored, untracked, tag, version, pin, stale-root, fixture, artifact, and manifest
negative canaries fail. The selected files are made read-only after
verification, and a final snapshot guard rejects any source mutation during the
run.

The manifest records source and dependency identities, host and toolchain,
build profile and sanitizer state, extension/load identity, connector and
fixture digests, public behavior identity, and artifact size and SHA-256. Local
artifact bytes are not compared across workspace paths.

## Sanitizer evidence

On a native Linux amd64 Docker host, use the outer launcher so the inspected
image digest, detached clean source snapshot, retained compile database, inner
manifest, and artifact are bound together:

```sh
scripts/run-linux-sanitized-cell.sh \
  "$PWD/.build/linux-amd64-sanitized"
```

The repository workflow `.github/workflows/linux-amd64-sanitized.yml` shows the
authoritative invocation and retains its output for handoff. The launcher itself
must observe Linux `x86_64`; `--platform linux/amd64` in Docker Desktop or a
devcontainer on Apple Silicon is emulation and is not native sanitizer evidence.
It also rejects endpoint overrides and requires the local `default` Unix-socket
Docker context to report a Linux `x86_64` daemon. Every Docker operation is
pinned to the captured socket, and the daemon ID, OS, and architecture must be
unchanged after the container exits, preventing a mutable client context from
silently delegating the build elsewhere. The launcher pulls and inspects the
exact digest, mounts a detached tagged source snapshot read-only, and emits an
anchored outer envelope over the inner outputs.
The inner gate rejects dirty or different source, non-native architecture,
wrong base OS, existing or overlapping roots, missing source pins, or a
production compile command without both sanitizer flags. It runs the private
scenarios and SQLLogicTests with leak detection and fail-fast ASan/UBSan
settings, inventories the loadable artifact, and retains both the compile
database and canonical sanitizer-flags report so custody can re-run the oracle.

After the product and sanitizer cells have both completed, bind the two
anchored manifests to the same source, dependency, and content identity:

```sh
scripts/verify-0.1-evidence-set.py \
  /product/evidence-root \
  /sanitizer/evidence-root \
  "$PWD"
```

The inner `run-linux-sanitized.sh` command produces measurements only. A direct
invocation cannot produce the required inspected-image launcher envelope and is
therefore rejected by the evidence-set verifier.

## Two-workspace handoff

From the tagged canonical repository:

```sh
scripts/reproduce-0.1-release.sh /absolute/new/reproduction-root
```

This creates two local cache-empty clones with distinct homes, temporary
directories, disabled pip caches and configuration, build roots, and evidence
roots. Each independently runs the authoritative product gate and retains its
gate log. The comparison ignores workspace-dependent compiler paths and
artifact bytes while requiring the source, dependency, toolchain, extension,
connector, fixture, SQL surface, compatibility result, and behavior identities
to match.

## Evidence custody

Preserve the selected artifact, both `manifest.json` files and SHA-256 anchors,
the retained sanitizer compile database and report, the sanitizer launcher
envelope and anchor, the evidence-set verification output, and gate logs
together in the product manager's chosen durable evidence store. The workflow's
90-day Actions artifact is a transfer mechanism, not durable release custody. A
co-located checksum detects accidental or one-sided mutation; durable external
custody is what anchors it against replacement of both files. Do not replace an
artifact under an existing manifest or move the `v0.1.0` tag. Rollback selects
an artifact reproduced from a prior immutable source commit; it never relabels
a `0.1.0` artifact.
