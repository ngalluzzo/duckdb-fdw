# 0.1.0 release evidence runbook

This runbook transfers the accepted first-query build, compatibility, and
release-evidence practice to Query Experience. RFC 0001 and the propagated
architecture, connector, and runtime contracts define behavior; this document
defines how to reproduce its evidence.

## Evidence cells

The supported product cell is DuckDB 1.5.4 at commit `08e34c447b`, platform
`osx_arm64`, on macOS 26.5.1 Apple Silicon arm64 with Apple clang 17 in C++11
mode, CMake 4.1.2, and Ninja 1.13.0. The artifact is source-built and loaded
directly with unsigned extension loading enabled.

`linux_amd64-sanitized` is a safety-evidence cell, not a supported product
platform. It runs the same production sources under AddressSanitizer and
UndefinedBehaviorSanitizer on a native Linux x86_64 executor inside the exact
Ubuntu image digest recorded in `release/0.1.0/pins.json`.

## Pre-tag product evidence

Use a path that does not exist. The runner refuses reused build state and
bootstraps checksummed CMake and Ninja binaries:

```sh
scripts/run-native-product-tests.sh /absolute/new/build-root release
```

The runner verifies source identities, fetches the exact DuckDB template and
submodule commits and trees, compiles the loadable and static test targets,
runs the private contract host and SQLLogicTests, rejects leaked test controls,
and loads the artifact in the pristine pinned Python host from an isolated
directory.

## Authoritative product gate

After the reviewed release commit is tagged exactly `v0.1.0`, run:

```sh
scripts/run-0.1-release-gate.sh \
  /absolute/new/build-root \
  /absolute/new/evidence-root
```

The gate requires a clean worktree and `v0.1.0^{commit} == HEAD`, checks every
version and content identity, runs the product suite from a fresh root, proves
that DuckDB 1.5.3 rejects the artifact before registration, writes and
re-verifies the artifact/manifest/checksum set, and observes the dirty-source,
untracked-source, tag, version, stale-root, fixture, artifact, and manifest
negative canaries fail. The selected files are made read-only after
verification.

The manifest records source and dependency identities, host and toolchain,
build profile and sanitizer state, extension/load identity, connector and
fixture digests, public behavior identity, and artifact size and SHA-256. Local
artifact bytes are not compared across workspace paths.

## Sanitizer evidence

On a native Linux amd64 executor, use the outer launcher so the actual pulled
image digest, clean source commit, and inner manifest are bound together:

```sh
scripts/run-linux-sanitized-cell.sh \
  "$PWD/.build/linux-amd64-sanitized"
```

The repository workflow `.github/workflows/linux-amd64-sanitized.yml` shows the
authoritative invocation. The launcher pulls and inspects the exact digest; the
inner gate rejects an unverified launcher identity, dirty or different source,
non-native architecture, wrong base OS, existing build root, missing source
pin, or production compile command without both sanitizer flags. It runs the
private scenarios and SQLLogicTests with leak detection and fail-fast
ASan/UBSan settings, then inventories the loadable artifact.

After the product and sanitizer cells have both completed, bind the two
anchored manifests to the same source, dependency, and content identity:

```sh
scripts/verify-0.1-evidence-set.py \
  /product/manifest.json \
  /product/manifest.sha256 \
  /sanitizer/manifest.json \
  /sanitizer/manifest.sha256 \
  "$PWD"
```

The inner `run-linux-sanitized.sh` command is not authoritative when invoked
directly; it deliberately requires identities supplied by the verified outer
launcher.

## Two-workspace handoff

From the tagged canonical repository:

```sh
scripts/reproduce-0.1-release.sh /absolute/new/reproduction-root
```

This creates two local cache-empty clones with distinct homes, caches, build
roots, and evidence roots. Each independently runs the authoritative product
gate. The comparison ignores workspace-dependent compiler paths and artifact
bytes while requiring the source, dependency, toolchain, extension, connector,
fixture, SQL surface, and behavior identities to match.

## Evidence custody

Preserve the selected artifact, both `manifest.json` files and SHA-256 anchors,
the evidence-set verification output, and gate logs together in the product
manager's chosen durable evidence store. A co-located checksum detects
accidental or one-sided mutation; durable external custody is what anchors it
against replacement of both files. Do not replace an artifact under an
existing manifest or move the `v0.1.0` tag. Rollback selects an artifact
reproduced from a prior immutable source commit; it never relabels a `0.1.0`
artifact.
