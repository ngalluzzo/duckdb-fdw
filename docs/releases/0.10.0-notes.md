# duckdb-fdw 0.10.0 release notes

Version 0.10.0 introduces the third REST pagination strategy, `response_next`,
accepted by [RFC 0016](../rfcs/0016-decide-body-signaled-rest-pagination.md).
A connector author can now declare a body-signaled continuation URL path
(`next_url_path`) that Runtime reads from the decoded JSON response — for
example `info.next` — reusing the same reconstruct-and-verify safety model
`link_next` already enforces for HTTP `Link: rel=next` headers.

## Declare body-signaled REST pagination

A relation whose API embeds the next-page URL in the response body (the common
`info.next` / `info.prev` shape) now declares:

```yaml
pagination:
  strategy: response_next
  dependency: sequential
  consistency: mutable
  target_scope: exact_operation_origin_and_path
  next_url_path: $.info.next
  page_size_parameter: page_size
  page_size: 20
  page_number_parameter: page
  first_page: 1
  page_increment: 1
  max_pages_per_scan: 42
```

The `next_url_path` field is the only structural difference from `link_next`.
Runtime extracts the value at that JSON path during the same single decode
pass that produces relation rows, validates the extracted URL against the
exact operation origin, path, and query-parameter multiset
(`ValidateNextTarget`), and reconstructs the next request locally — the
continuation URL is a verified signal, never a dereferenced fetch target.
A JSON `null` or absent path means "no next page."

`EXPLAIN` reports the new `response_next` strategy value alongside the
existing `disabled`, `link_header`, and `graphql_cursor`.

## Compatibility

`response_next` is additive: no existing accepted package changes behavior.
The `duckdb_api/v1` JSON schema's REST pagination `oneOf` gains a third
branch (`responsePagination`), and the `1.0.0` candidate freeze
([`release/1.0.0/freeze.json`](../release/1.0.0/freeze.json)) graduates
`response_next` from `accepted_candidate_revisions` into the schema-closed
`pagination_strategies.rest` set (now `{disabled, link_next, response_next}`).

The broader body-URL/offset/cursor pagination category remains a permanent
`1.0.0` exclusion: only the reconstruct-and-verify shape named
`response_next` is in the closed set. Numeric offset or page-number-only
traversal (without the matching origin/path/page-progression guarantee),
cursor-in-body strategies, and reverse or bidirectional traversal still
require their own later accepted RFC.

This release is a backward-compatible capability addition under the pre-1.0
versioning model; it is a `0.Y.0` minor rather than a `0.9.x` patch.

## `1.0.0` release gate

The `1.0.0` public contract candidate freeze no longer treats `response_next`
as a pending candidate revision. `release/1.0.0/freeze.json` records
`pagination_strategies.rest = ["disabled", "link_next", "response_next"]`,
and the `accepted_candidate_revisions` list is empty.

The `1.0.0` release itself remains gated on supporting at least 10 different
API providers (two exist as of this release: GitHub and Rick and Morty).

## Build and verification

On the supported product cell:

```sh
make bootstrap
make build
make test
make demo
```

`make demo` loads the permanent GitHub package and executes its anonymous
generated relation. `make test` uses controlled local HTTP and GraphQL services
for compiler, catalog, planning, request, decoding, relational, cancellation,
reload, and lifecycle correctness, including the new `response_next`
AdvanceBody differential tests and BaseDomain property test.

## Known limitations

- No package in the repository declares `response_next` yet. The
  `connectors/rickandmorty` `character_search` relation is the natural
  candidate, but its API omits a `page_size` query parameter; the
  `response_next` validator (shared with `link_next`) checks that all
  declared query parameters appear in the continuation URL. Resolving this
  tension (either by relaxing the query-multiset check for `response_next`
  or by making `page_size` optional for fixed-page-size APIs) is a follow-on.
- All limitations from [0.9.0](0.9.0-notes.md#known-limitations) remain.
