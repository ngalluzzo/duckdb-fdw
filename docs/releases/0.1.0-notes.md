# DuckDB API 0.1.0 release notes

## Status

`0.1.0` is the first fixture-backed native preview. It builds and runs the
accepted first-query behavior on one exact compatibility cell; the support
claim does not extend beyond that cell or the behavior documented here.

## What works

- A pinned source build produces the native `duckdb_api` extension.
- Direct unsigned loading into the pinned DuckDB 1.5.4 Python host exposes
  `duckdb_api` version `0.1.0` and one table function.
- The fixture-backed query:

  ```sql
  SELECT id, name, active
  FROM duckdb_api_scan(
      connector := 'example',
      relation := 'items'
  )
  ORDER BY id;
  ```

  returns `(1, 'alpha', true)`, `(2, 'beta', false)`, and
  `(3, 'gamma', true)` with `BIGINT`, `VARCHAR`, and `BOOLEAN` columns.
- Private deterministic oracles cover offline bind and planning, bounded
  batches, strict conversion, redacted decode and schema failures,
  cancellation, repeated scans, and teardown.
- An unknown relation fails during bind with an actionable identifier-only
  diagnostic.

Run the product-facing example from the repository root:

```sh
make demo
```

Use `make help` for the complete source-development target list. `make build`,
`make test`, `make demo`, and `make paths` reuse pinned developer state by
default. `PROFILE=release` selects a release-profile developer build, and
`DUCKDB_API_DEV_ROOT=/absolute/path` relocates that state.

## Supported compatibility cell

| Component | Supported value |
| --- | --- |
| DuckDB | 1.5.4, commit `08e34c447b` |
| DuckDB platform | `osx_arm64` |
| Host | macOS 26.5.1 on Apple Silicon arm64 |
| Compiler | Apple clang 17.0.0, C++11 |
| Build tools | CMake 4.1.2 and Ninja 1.13.0 |
| Python host | Python 3.14 with the hashed DuckDB 1.5.4 wheel |
| Load mode | Direct unsigned loading of a local source-built artifact |

No other compatibility cell is supported. The separately pinned Linux amd64
ASan/UBSan cell supplies safety evidence only; it is not a product platform.

## What does not work yet

- Live HTTP or GraphQL requests.
- Authentication, secrets, user-supplied URLs, redirects, proxies, pagination,
  retries, caching, or provider plugins.
- Arbitrary connector installation, registration, reload, or stable connector
  authoring compatibility.
- Predicate, ordering, limit, or offset pushdown, or custom explain output.
- Binary distribution, installation, signing, automatic updates, or
  compatibility with a system DuckDB installation.

The template's developer DuckDB CLI statically links `duckdb_api` for testing.
It is not a clean host and must not be used as evidence that the loadable
artifact works. The first-query demo therefore uses the separate pinned Python
host.

## Developer checks versus release evidence

Reusable developer artifacts are intentionally convenient:

```sh
make build
make test
make demo
make paths
```

`make verify` runs the complete pre-tag product suite from a fresh build root
instead of reusing developer state. It is useful development evidence, but it
is not the authoritative release gate.

After the reviewed source is tagged exactly `v0.1.0`, release evidence must
also include the clean tagged product gate, the pinned native Linux sanitizer
cell, manifest and checksum verification, durable custody, and two cache-empty
workspace reproductions. Follow the
[0.1.0 release evidence runbook](0.1.0.md); the release tag, manifests, and
checksums are the authoritative identities for published `0.1.0` bytes.
