# duckdb-fdw 0.18.0 release notes

Version 0.18.0 adds bounded Runtime admission and bulkhead isolation, accepted
by [RFC 0026](../rfcs/0026-bound-runtime-admission-and-bulkheads.md). Existing
v1, v2, and v3 package bytes, SQL functions, plans, and successful single-scan
results are unchanged; the extension now places fixed local ceilings on
concurrent work.

## Executor-local capacity

Each HTTP executor owns one admission service for credential resolution,
active scans, in-flight requests, ordinary retry and rate-limit waiters,
buffered request/response/decode bytes, and decoded rows. It checks every
applicable global, connector, destination, opaque credential-principal, and
exact operation bulkhead limit atomically. No caller can retain a partial
outer permit while waiting for another dimension.

Provider, scan, and request queues are finite, cancellable, deadline-aware,
and FIFO within an exact key. An older ticket blocked by its own saturated key
does not prevent an independently eligible connector from progressing. Buffer
and recovery-wait shortages fail immediately so the rejected work retains no
request, response, or waiter state.

## Diagnostics and lifecycle

Local overload is reported as a `resource` failure with the distinct
`local_admission` class and closed reason, scope, limit, observed, requested,
and wait-count facts. It is never classified as a remote timeout, never
retried, and never consumes a transport attempt. Identity values, hashes,
destinations, credentials, queue keys, tickets, and timestamps remain absent
from diagnostics.

The DatabaseInstance lifecycle closes Query publication, Runtime generation
admission, and then the shared executor. Executor close wakes every queued
provider, scan, and request, closes quota coordination, and rejects future
authority. Live streams and move-only reservations retain release-safe shared
state; repeated close is harmless.

## Compatibility and exclusions

- No connector syntax, package migration, SQL setting, environment variable,
  or per-scan input changes the installed limits.
- V1 remains one-attempt; v2 retry and v3 reactive rate-limit semantics keep
  their existing replay and waiting ceilings.
- Admission is local to one executor and DuckDB DatabaseInstance. It is not a
  process-global or distributed scheduler.
- Circuit breaking, failure history, open/half-open state, caching,
  single-flight, and parallel pagination remain excluded.

## Evidence

Deterministic tests cover every limit and scope, exact-key FIFO and eligible
bypass, starvation resistance, cancellation/deadline/close races, ticket
exhaustion, retry/rate waiter caps, response release before recovery waits,
attempt-not-charged ordering, byte/row and multi-batch ownership, and
release-safe destruction. Boundary tests also prove curl retained metadata,
GraphQL request bodies, and cursor continuation storage are admitted before
growth. Actual DuckDB tests hold authenticated REST and
GraphQL work at one saturated GitHub bulkhead, prove a second same-key scan is
rejected before transport with exact safe counts, and complete an unrelated
Rick and Morty query through the same executor with `threads=1`. A separate
named v3 REST scenario keeps one slow scan live while another traverses
ordinary retry and a measured rate-limit wait, and proves the unrelated query
completes before recovery.
