# duckdb-fdw 0.5.0 release notes

Version 0.5.0 proves that the permanent native extension can turn one fixed,
authenticated, multi-page HTTP resource into a bounded DuckDB relation. It is
a source-built preview for the exact supported product cell, not a published
binary extension or a general connector-package system.

## Product surface

Create the existing explicitly named temporary secret, then query the new
relation:

```sql
CREATE TEMPORARY SECRET github_default (
    TYPE duckdb_api,
    PROVIDER config,
    TOKEN 'github-token-value'
);

SELECT id, full_name, private, fork, archived
FROM duckdb_api_scan(
    connector := 'github',
    relation := 'authenticated_repositories',
    secret := 'github_default'
)
ORDER BY id;
```

The scan begins with the fixed GitHub `GET
/user/repos?per_page=100&page=1` operation. It follows only a strictly validated
sequential next-page transition for the same origin and path, reconstructs each
request from the immutable plan, and reuses the scan-scoped bearer capability.
Every successful pull returns a nonempty typed batch; only absence of a next
page produces clean exhaustion.

The result is a duplicate-preserving bag from a mutable source. Remote ordering
and snapshot consistency are not product guarantees. DuckDB still owns filter,
ordering, limit, and offset, so a local `ORDER BY` is required for deterministic
presentation.

## Hard execution envelope

- One attempt per page; at most 32 pages and 32 requests.
- At most 16 KiB response headers, 8 MiB wire body, 8 MiB decompressed body,
  and 100 records per page.
- At most 512 KiB response headers, 64 MiB wire body, 64 MiB decompressed body,
  and 3,200 records per scan.
- At most 512 bytes per extracted string, 16 JSON nesting levels, 2 MiB retained
  decoded-page memory, 64 output rows per batch, one active request, and 30
  seconds per scan.

An invalid or escaping transition, exhausted budget, cancellation, early close,
HTTP failure, decode/schema failure, or late-page failure is terminal. A query
never converts such a condition into a successful complete-looking partial
relation.

## Preserved behavior and exclusions

The anonymous `github.duckdb_login_search_page` and authenticated
`github.authenticated_user` relations retain their 0.4.0 behavior and narrower
single-response budgets. Bind, `DESCRIBE`, `EXPLAIN`, and `PREPARE` remain
offline. Credentials still come only from an explicit temporary
`duckdb_api/config` secret resolved at execution.

This release adds no retries, rate-limit waits, parallel pages, resume state,
caller-selected URL/header/page inputs, cache, provider expansion, GraphQL
execution, connector YAML loading, package distribution, or public native ABI.

## Build and verification

On the exact supported cell, `make build` creates the unsigned local artifact,
`make test` runs the cached developer oracles, and `make verify` runs the fresh
product gate. `make paths` reports the artifact and pinned Python host used by
the interactive examples. The public GitHub check is compatibility evidence;
the private controlled three-page service is the deterministic correctness
oracle.

The exact DuckDB, toolchain, system libcurl, source graph, controlled
composition, and public contract identities are recorded in
`release/0.5.0/pins.json`.
