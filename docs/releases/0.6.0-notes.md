# duckdb-fdw 0.6.0 release notes

Version 0.6.0 makes ordinary relational composition trustworthy over the
existing authenticated GitHub repository relation. It does not broaden the
SQL surface or add another remote optimization: DuckDB still owns the final
filter, projection, ordering, limit, and offset.

## Product surface

The existing temporary secret and table function are unchanged. A query may
compose the one supported remote restriction with DuckDB-local work:

```sql
CREATE TEMPORARY SECRET github_default (
    TYPE duckdb_api,
    PROVIDER config,
    TOKEN 'github-token-value'
);

SELECT full_name
FROM duckdb_api_scan(
    connector := 'github',
    relation := 'authenticated_repositories',
    secret := 'github_default'
)
WHERE visibility = 'private'
  AND archived = FALSE
ORDER BY id DESC
LIMIT 2;
```

The structured `visibility = 'private'` conjunct may add
`visibility=private` to every repository request. The mapping is a remote
superset optimization, so DuckDB retains and evaluates the complete filter,
projects the result, orders surviving rows, and applies the limit. `OR`, `NOT`,
unsupported expressions, other visibility values, and unavailable adapter
metadata use the unrestricted traversal when DuckDB executes the scan.

## Explanation and planning

`EXPLAIN` reports the typed predicate candidate, remote predicate and accuracy,
retained filter scope, complete projection closure, relational owners,
undelegated ordering and bounds, adapter capability state, and one structured
classification category and reason. Explanation is safe metadata: it performs
no secret resolution, request construction, or network I/O and is never parsed
for execution authority.

Planning distinguishes four normal outcomes:

- `Exact` and `Superset` select a Connector-validated typed conditional input.
- `Unsupported` retains an unrestricted remote plan.
- `Ambiguous` also retains the unrestricted plan, while identifying that more
  than one safe input remained possible.
- An invalid declaration or operation decision fails deterministically before
  Runtime entry instead of being mislabeled as a normal fallback.

Exact and ambiguous classifications are proven through non-installable
Connector fixtures. The installed catalog still exposes only the reviewed
GitHub visibility superset mapping.

Base-operation selection is deterministic before predicate classification.
Eligible non-fallback operations are ranked by satisfied selector specificity
and then priority; the sole fallback is considered only when none is eligible,
and a top-rank tie fails planning. Controlled fixtures prove the winner,
fallback, tie, contradictory-selector, and multiple-fallback cases without
adding an installed operation.

## Correctness evidence

The installed-path actual-DuckDB oracle compares identical optimized and
forced-local SQL over `TRUE`, `FALSE`, and `NULL` outcomes and duplicate
occurrences. Product tests cover complete and selective traversal, unsupported
and unencodable fallback, total ordering, tie groups under non-total ordering,
local limits and offsets, prepared-statement copies, early close,
cancellation, and DuckDB scan pruning with zero Runtime opens. A separate
production-planner law oracle uses Connector-owned non-installed fixtures for
Exact, Ambiguous, and invalid outcomes, because those states intentionally
grant no installed Runtime authority.

Runtime admits only coherent immutable plans. The installed GitHub Superset
plan has one typed selected-input path, while Unsupported and Ambiguous use the
same unrestricted path. Controlled Exact remains planner-only evidence, and an
Exact relabeling of the GitHub source fails before credentials or network I/O.
Existing pagination, credential, network, resource, cancellation, and close
guarantees remain classification-independent across executable forms.

## Preserved exclusions

This release adds no remote projection, ordering, limit, or offset pushdown; no
second public predicate mapping; no SQL-text reconstruction; and no connector
YAML loading, retry, cache, provider, GraphQL, caller-selected request, or new
credential/network authority. It remains a source-built preview for the exact
supported product cell rather than a published binary extension or stable
native ABI.

## Build and verification

On the supported cell, `make build` creates the local unsigned artifact,
`make test` runs the cached developer oracles, `make demo` exercises the public
GitHub path, and `make verify` performs a fresh native product build and
controlled direct-load contract. `make paths` reports the exact pinned Python
host, loadable artifact, and statically linked test CLI.

The DuckDB, toolchain, system libcurl, complete source graph, controlled
composition, and public contract identities are recorded in
`release/0.6.0/pins.json`.
