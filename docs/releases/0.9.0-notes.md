# duckdb-fdw 0.9.0 release notes

Version 0.9.0 is the public connector authoring and API candidate release
described in `ROADMAP.md`. It introduces no public declaration, SQL behavior,
lifecycle mechanism, or compatibility surface: the `duckdb_api/v1` package
workflow from 0.8.0 is unchanged. This release validates that workflow across
independently authored packages, removes the deprecated 0.7.0 dispatcher on
schedule, and freezes the intended `1.0.0` public API candidate for
compatibility testing.

## Load a package and query it

Load the unsigned extension, then pass the absolute path of a local package
root:

```sql
LOAD '/absolute/path/to/duckdb_api.duckdb_extension';

CALL duckdb_api_load_connector(
    package_root := '/absolute/path/to/duckdb-fdw/connectors/github'
);

SELECT id, login, site_admin
FROM github_duckdb_login_search_page();
```

The load result contains the connector identifier, package version, spec
version, semantic digest, relation count, and a `changed` flag. Publication is
atomic: compilation, validation, naming, or coordination failure publishes no
function and does not alter an active connector.

An accepted relation is registered in `system.main` as
`<connector_id>_<relation_id>`. Declared relation inputs become named DuckDB
arguments. An authenticated relation also receives the reserved required
`secret VARCHAR` argument; the secret value itself is never part of package
source or a compiled plan.

## Maintained GitHub package

The repository includes a complete package at `connectors/github`:

| Function | Protocol | Authentication | Shape |
| --- | --- | --- | --- |
| `github_duckdb_login_search_page()` | REST | Anonymous | Up to three required user rows |
| `github_authenticated_user(secret := ...)` | REST | Bearer | Exactly one current-user row on success |
| `github_authenticated_repositories(secret := ...)` | REST | Bearer | Bounded sequential Link-pagination bag |
| `github_viewer_repository_metrics(secret := ...)` | GraphQL | Bearer | Bounded sequential cursor-pagination bag |

Create a temporary secret before executing an authenticated function:

```sql
CREATE TEMPORARY SECRET github_default (
    TYPE duckdb_api,
    PROVIDER config,
    TOKEN '<short-lived-github-token>'
);

SELECT full_name, stars, primary_language, updated_at
FROM github_viewer_repository_metrics(secret := 'github_default')
WHERE archived = FALSE
ORDER BY stars DESC, full_name
LIMIT 10;
```

Package loading, bind, `DESCRIBE`, `EXPLAIN`, and `PREPARE` do not perform
network I/O or resolve credentials. Runtime acquires the named temporary secret
only when execution needs the package-declared authenticator.

## Reload and introspection

The extension retains the canonical local root and immutable active generation
for each loaded connector. Reload recompiles that root and publishes only a
compatible changed generation:

```sql
CALL duckdb_api_reload_connector(connector := 'github');
```

An identical package returns `changed = false`. An invalid or incompatible
reload leaves the active generation, its registered functions, and already
bound or executing scans usable.

These read-only functions expose package metadata without roots or secrets:

```sql
SELECT * FROM duckdb_api_loaded_connectors();
SELECT * FROM duckdb_api_loaded_relations();
SELECT * FROM duckdb_api_relation_arguments();
```

## Connector-package contract

The v1 package is a closed, failsafe YAML contract. It covers:

- connector identity and SemVer;
- static typed relation outputs and named inputs;
- anonymous or declared bearer authentication;
- exact HTTPS origins, paths, fixed headers, and resource ceilings;
- REST root-object, root-array, or terminal-collection responses;
- disabled, sequential Link, or sequential GraphQL cursor pagination;
- conservative declared predicate mappings with offline occurrence fixtures;
- query-only GraphQL operation name, fixed arguments, selection, pagination,
  partial-data policy, and request-body bounds.

Unknown, ambiguous, unsafe, incomplete, or over-budget forms are rejected.
Diagnostics identify the package-relative file, line and column when known,
and YAML path while redacting the canonical root, source scalar values,
credentials, request bodies, and provider responses.

The compiler produces one immutable generation containing package identity,
registration metadata, relation declarations, operation recipes, policies,
and fixture identity. Query, planning, and Runtime consume bounded generated
views; they do not parse YAML or reconstruct package meaning.

## Relational and runtime behavior

DuckDB remains the owner of relational correctness. The planner chooses only a
remote operation and restriction proven safe for the complete typed request.
Unsupported, ambiguous, `NULL`, over-budget, or capability-unavailable
predicate structure uses conservative fallback, and every required residual is
evaluated exactly once by DuckDB. Limits and offsets are applied only after
required filtering and ordering.

Runtime accepts only the scheme, host, port, request structure, authenticator,
and budgets carried by the immutable plan. Connector policy can narrow host
policy but cannot widen it. REST and GraphQL traversal is sequential unless
independence and consistency are proven. Malformed data, GraphQL errors, unsafe
page transitions, cancellation, or resource exhaustion fail the statement
rather than returning a successful partial result.

Active scans retain the exact package generation they observed. Reload and
connection shutdown cannot replace their declaration or free it prematurely;
new scans observe the newly committed catalog generation.

## Removal of the 0.7.0 dispatcher

Accepted RFC 0012 scheduled `duckdb_api_scan` for removal in 0.9.0, before the
API candidate freezes. That removal is complete: the function no longer
registers when the extension loads, and its dispatcher-only diagnostics
(`unknown connector identifier`, `unknown relation identifier`, `required
named argument relation is missing`, and rejection of a `secret` argument on
an anonymous relation) no longer apply, since a generated function's identity
is fixed at registration and carries no runtime connector or relation lookup.
The 0.8 package-form migration table in the
[0.8.0 release notes](0.8.0-notes.md#migration-from-070) remains the
authoritative mapping from the removed 0.7 calls.

## Package-independence proof

A second, independently authored `duckdb_api/v1` package (a Rick and Morty API
connector, test-only and not distributed with the extension) proves that the
compiler, planner, and generated-function surface depend only on the v1
contract: two unrelated packages compiled through the same pipeline produce
equivalent, independently correct output with no cross-package coupling.

## 1.0.0 API candidate frozen

The public SQL surface, diagnostics vocabulary, connector-package contract,
and compatibility boundary intended for `1.0.0` are enumerated and frozen in
`release/1.0.0/freeze.json` (see `release/1.0.0/freeze.md` for the narrative).
Freezing the candidate is not shipping it: the remaining `1.0.0` gates are a
green declared DuckDB support matrix and a final release audit. No further
public surface change is expected before `1.0.0`.

## Build and verification

On the supported product cell:

```sh
make bootstrap
make build
make test
make demo
```

`make demo` loads the permanent GitHub package and executes its anonymous
generated relation. `make test` uses controlled local HTTP and GraphQL services
for compiler, catalog, planning, request, decoding, relational, cancellation,
reload, and lifecycle correctness. Live GitHub examples are compatibility
probes only and use privacy-safe output.

`make verify` performs the complete product suite from a fresh build root.
`make paths` prints the exact pinned Python host, static test CLI, loadable
artifact, and build locations.

## Known limitations

- Packages load only from an explicit absolute local directory. There is no
  package discovery, registry, remote installation, update service, signature,
  or trust-policy distribution mechanism.
- The extension remains an unsigned source build for the exact DuckDB 1.5.4,
  Python 3.14, Apple Silicon macOS product cell documented in the README.
- Authentication is limited to explicitly named temporary bearer secrets.
  Persistent and environment-backed providers, implicit selection, and OAuth
  are not supported.
- There are no retries, rate-limit waits, parallel page requests, prefetch,
  resume state, or caching.
- SQL cannot provide request URLs, headers, GraphQL documents, destinations,
  pagination values, or credential values. Those authorities remain in the
  validated package and host policy.
