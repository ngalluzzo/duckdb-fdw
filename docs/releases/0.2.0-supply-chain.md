# 0.2.0 Community supply-chain evidence

This maintainer runbook covers the first bounded Engineering Enablement
provider slice for Community delivery. Query Experience owns the stock-DuckDB
install/load/query oracle, exact support matrix, failure guidance, and public
release acceptance. Nothing here is ordinary-user installation guidance.

## What the slice proves

`audit_dependencies.py` operates offline over caller-supplied local Git
repositories. It verifies each exact commit/tree, the candidate's root MIT
license, DuckDB and extension-template MIT bytes, and the deliberate absence of
a root license in the two external-only repositories. If either repository
gains license evidence, admission stops so its role and obligations can be
reviewed rather than preserving a stale absence assumption.

The audit admits the tracked dependency expectations only when their canonical
bytes match the SHA-256 bound into the candidate-cycle pins. Every local
upstream name is one fixed path component and therefore a direct child of the
upstreams root. Git replacement objects and lazy object fetching are disabled,
so local refs, checkout state, replacement refs, and promisor remotes cannot
reinterpret the admitted commit objects.

The resulting `dependency-audit.json` is an input-admission record, not the
complete release license audit. It explicitly defers per-platform compiler and
operating-system dependencies, the actual Community artifact inventory, and
dependencies or notices discoverable only from those outputs. Local macOS
observation of only `libc++` and `libSystem` linkage is useful evidence for that
one local build, not a claim about Community platform rows.

`verify_candidate.py` receives an exact source commit. It reads the commit and
tree directly, requires a sole top-level `duckdb_extension_load(duckdb_api ...)`
command with version `0.2.0`, rechecks the root MIT bytes, and requires exactly
the pinned `duckdb` and `extension-ci-tools` gitlinks. The commit's regular
`.gitmodules` blob must contain exactly those two modules with their official
URLs, expected paths, and branches. The command then binds the anchored
dependency record and its exact expectation digest and records the tracked
descriptor expectation. It never infers content from `HEAD`, the index, the
worktree, or ambient Git configuration, and never creates a source ref,
artifact, custody record, platform row, or support claim.

The tracked descriptor expectation has no authority. Its source ref, source
commit, and maintainers remain unset, and its publication status is
`not_submitted`; those exact bytes remain bound into the candidate record.
The separate tracked `description.yml` proposal names the published immutable
candidate commit `47dc6169ae820f70beb0c2722b8a8f5288cd1469`, the approved maintainer,
and the exact MIT/C++/cmake project metadata. It contains no platform exclusion,
documentation block, or support claim.

The reviewed `descriptor-cycle.json` authorizes exactly one handoff by binding
the published source commit/tree, proposal, pins, pending expectation, and the
exact candidate and dependency record plus anchor digests. The adjacent anchor
files remain custody evidence; a caller cannot authorize an alternate source
by regenerating a consistent candidate, dependency audit, proposal, and set of
co-located checksums.

`verify_descriptor.py` re-admits the complete anchored candidate and dependency
records against the reviewed cycle and pins before parsing the proposal with a
deliberately small canonical YAML grammar. It rejects duplicate or ambiguous
YAML, mutable refs, jointly re-anchored substitutions, metadata drift, changed
custody anchors, and reused or linked boundaries. Its
anchored output records only local proposal admission and `not_submitted`;
Community review, CI, signing, deployment, artifacts, platform evidence,
publication, and support remain external gates.

The next provider boundary is `collect_build_evidence.py`. It consumes only
explicit local canonical exports and downloaded bytes. The code-reviewed
`build-authorities.json` registry is empty in this change, so no current PR or
run—including PR 2256—is admitted yet. Maintainer approval must bind the exact
descriptor admission, pins, extension source, Community descriptor-PR head,
base, workflow, run execution head, attempt, and five export digests before the
command can normalize a run. The PR head is a commit in the Community
repository or its fork; it is not the `duckdb_api` source commit named by the
descriptor. This prevents a
caller from treating mutable GitHub display state or a self-consistent local
export as authority.

The collector separately binds the reviewed expected matrix, including
`excluded_unclaimed` combinations that create no GitHub job, log, runner, or
artifact. It then requires the complete GitHub job `total_count`, retains every success,
failure, cancellation, skip, and other terminal conclusion, preserves raw
matrix and runner labels, verifies exact DuckDB identity for artifact-producing
jobs, and inventories exactly one downloaded log per job. Every declared
artifact archive is checked against its approved name, producing job, size,
SHA-256, run, and attempt; missing, extra, expired, duplicated, linked, or
changed downloads stop admission. Artifact and log roots are held through
non-following directory descriptors, so a concurrent path replacement cannot
redirect later entry opens; root identity, entries, and inventory must remain
stable through the final byte check. The per-job and aggregate records remain
local provenance evidence only. They do not establish signing, deployment,
stock-host lifecycle success, compatibility, platform exclusions, or support.

The successful native Actions archive, its unsigned extension, the served
Community gzip, and its signed extension deliberately have different
complete-byte digests. The unsigned extension ends in a 256-byte all-zero
signature placeholder; deployment preserves the preceding bytes, replaces
that block with the Community signature, and gzip-compresses the result.
Post-merge deployment evidence must retain all four identities and the shared
pre-signature payload digest. Exact byte derivation does not validate the
signature: only a stock DuckDB install/load under default signature policy can
establish that acceptance for a prospective support row. The separate Wasm
artifact/Brotli path remains unclaimed by this native release contract.

## Run and inspect

Follow the exact commands in
`release/0.2.0/enablement/RUNBOOK.md`. Preserve `dependency-audit.json`,
`dependency-audit.sha256`, `candidate.json`, `candidate.sha256`,
`descriptor-admission.json`, and `descriptor-admission.sha256` together.
After a real build authority is approved, also preserve all five canonical
exports, including `matrix.json`, the exact artifact/log download roots, every
per-job build record and anchor, and the aggregate build inventory and anchor
together.
The anchor format is one lowercase digest, two spaces, the JSON basename, and
one newline. The verifiers reject non-canonical JSON, changed anchors, mutable
source identities, replacement objects, reused output roots, symlink leaves,
license drift, version drift, unsafe upstream paths, changed dependency
expectations, and a dependency record produced for another candidate or pin
set. File diagnostics deliberately use stable logical labels rather than
caller-supplied filesystem paths.

Run the focused deterministic tests with:

```sh
bash scripts/test-community-enablement.sh
```

These tests use synthetic local Git repositories and network-free negative
fixtures. They do not establish Community build, signing, publication, hosted
custody, stock-host lifecycle, platform support, or release completion. Those
remain later delivery gates under Accepted RFC 0004.
