# duckdb-fdw 0.16.0 release notes

Version 0.16.0 adds bounded replay-safe retry for transient failures, accepted
by [RFC 0024](../rfcs/0024-enable-bounded-replay-safe-retry.md). A connector
author can opt a proved read-only REST or GraphQL operation into retry by
publishing a package-major `duckdb_api/v2` generation. Existing
`duckdb_api/v1` packages remain byte-compatible and perform one attempt.

## Opt in from a v2 package

An eligible operation may add:

```yaml
retry:
  max_attempts_per_step: 3
  max_delay_milliseconds: 100
  max_cumulative_waiting_milliseconds_per_scan: 250
```

All fields are required. Attempts must be `2..3`, delay `1..100`
milliseconds, and cumulative scan waiting `1..250` milliseconds. Adding or
changing retry behavior requires a package-major version because it changes
request amplification, resource use, and terminal-failure behavior. See the
complete package under [`examples/retry-v2`](../../examples/retry-v2).

## Safety boundary

Runtime retries only an unaccepted page from a declared replayable read. The
initial matrix contains closed zero-response transient transport failures and
complete `502`, `503`, or `504` responses without `Retry-After`. It excludes
`429`, server-directed waiting, partial responses, authorization and policy
failures, malformed responses, decode/schema/continuation failures,
cancellation, deadlines, writes, and unknown replay safety.

Each scan retains one credential authority and revision snapshot. Every
attempt performs fresh destination-policy validation and debits the same
aggregate attempts, bytes, waiting, memory, rows, pages, and deadline ledger.
Page and cursor state commit only after the complete response, decode, schema,
continuation, resource, and buffer acceptance boundary. Once rows from a page
are exposed, that page is never replayed.

Structured runtime diagnostics add the final attempt number, cumulative retry
delay, and step exposure state. The installed hard maxima are three attempts
per step, 96 attempts per scan, 100 milliseconds per delay, and 250
milliseconds cumulative waiting.

## Compatibility and migration

- V1 packages require no changes and keep one-attempt behavior.
- To opt in, change every package file to `api_version: duckdb_api/v2`, bump
  the connector package major version, and add `retry` only to proved safe
  read operations.
- Migrating to v2 without a `retry` block changes no request behavior.
- Rate-limit waiting, proactive quota scheduling, caching, parallel pages,
  resume, writes, and author-defined idempotency keys remain out of scope.

## Evidence

The controlled demonstration executes `503 -> pre-response connection reset
-> valid duplicate-bearing page` and produces the same occurrence bag as the
failure-free scan. The automated matrix covers retry classification,
page-atomic REST and GraphQL traversal, partial responses, malformed decode,
attempt exhaustion, credential identity, budget composition, deterministic
jitter bounds, cancellation during backoff, and final structured diagnostics.
