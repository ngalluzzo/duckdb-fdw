# duckdb-fdw 0.7.0 release notes

Version 0.7.0 adds the first GraphQL-backed product relation without changing
the DuckDB-facing function or granting caller-selected request authority. The
release keeps GraphQL in the intended v1 protocol set because the new relation
uses the same immutable planning, capability-scoped authentication, bounded
execution, typed batch, cancellation, and close boundaries as the existing
REST relations.

## Product surface

Create an explicitly named temporary secret and query the authenticated
viewer's repositories through ordinary SQL:

```sql
CREATE TEMPORARY SECRET github_default (
    TYPE duckdb_api,
    PROVIDER config,
    TOKEN 'github-token-value'
);

SELECT full_name, stars, primary_language, updated_at
FROM duckdb_api_scan(
    connector := 'github',
    relation := 'viewer_repository_metrics',
    secret := 'github_default'
)
WHERE archived = FALSE
ORDER BY stars DESC
LIMIT 10;
```

The fixed relation returns these columns:

| Column | DuckDB type | Value contract |
| --- | --- | --- |
| `id` | `VARCHAR` | Required opaque GraphQL node identifier |
| `full_name` | `VARCHAR` | Required repository name with owner |
| `owner_login` | `VARCHAR` | Required owner login |
| `stars` | `BIGINT` | Required stargazer count |
| `primary_language` | `VARCHAR` | Nullable language name |
| `private` | `BOOLEAN` | Required privacy flag |
| `archived` | `BOOLEAN` | Required archive flag |
| `updated_at` | `VARCHAR` | Required validated GraphQL `DateTime` serialization |

Rows are a duplicate-preserving bag from a mutable cursor traversal. The
relation promises neither a stable snapshot nor DuckDB-visible remote order;
use SQL `ORDER BY` when presentation order matters. DuckDB owns every filter,
projection, join, ordering, limit, and offset.

## Fixed GraphQL profile

The extension admits one repository-owned query document at
`https://api.github.com/graphql`. It requests `viewer.repositories` in pages
of 100 with the complete owner, collaborator, and organization-member
affiliations and a fixed `UPDATED_AT DESC` cursor enumeration. Callers cannot
provide a document, variable, endpoint, root field, selection, mutation,
introspection request, or pagination value.

Runtime recomputes the document digest during admission, constructs a compact
JSON body from the immutable document plus its stream-owned nullable cursor,
checks and debits that complete body before placing the bearer header, and
sends one `application/json` POST at a time. Traversal is limited to 32 pages
and 3,200 rows, with 8 KiB per-body and 256 KiB per-scan serialized-body
ceilings in addition to the existing response, decode, memory, header,
concurrency, and wall-time limits.

Any nonempty top-level GraphQL `errors` array fails the statement, including a
response that also contains data. Missing, duplicate, null-required, or
wrongly typed fields; invalid or repeated cursors; resource exhaustion;
cancellation; and later-page failures also fail terminally with redacted
diagnostics. A null `primaryLanguage` becomes SQL `NULL`; no sentinel value is
invented.

## Correctness evidence

Deterministic controlled services cover exact first and later request bodies,
zero-, one-, and full-page responses, duplicates, nullable language values,
strict scalar and envelope failures, GraphQL errors with and without data,
cursor transitions, page and body limits, no prefetch, cancellation
checkpoints, close and destruction, terminal error replay, and independent
streams. Actual DuckDB queries cover offline bind, `DESCRIBE`, `EXPLAIN`, and
`PREPARE`, typed null vectors, filters, ordering, limits, joins, repeated
execution, and retained REST behavior. A privacy-safe live probe confirms only
upstream compatibility and never records repository values, counts, cursors,
documents, variables, or credential material.

## Preserved exclusions

This release adds no connector-package or YAML GraphQL syntax, arbitrary
documents, generated selections, introspection, mutations, partial-data
recovery, remote relational pushdown, retries, rate-limit waiting, caching,
parallel page requests, prefetch, resume state, persistent credentials, or new
distribution mechanism. It remains a source-built preview for the exact
supported product cell rather than a published binary extension or stable
native ABI.

## Build and verification

On the supported cell, `make build` creates the local unsigned artifact,
`make test` runs the cached developer oracles, `make demo` exercises the
retained anonymous REST path, and `make verify` performs a fresh native product
build and controlled direct-load contract. `make paths` reports the exact
pinned Python host, loadable artifact, and statically linked test CLI.

The DuckDB, toolchain, system libcurl, complete source graph, controlled
composition, and public contract identities are recorded in
`release/0.7.0/pins.json`.
